Chirper / Social Media Notifications

Scope

Question: How does CS2's in-game "Chirper" social media system create, display, and manage chirp notifications -- the Twitter/X-like messages from citizens, brands, and city services?

Verdict: Chirps are ECS entities created by CreateChirpSystem from a NativeQueue that other systems enqueue into. Each chirp has a sender (citizen, brand, or service account), a like count that animates over time via a viral curve, and linked entities for clickable references. A deduplication system prevents the same chirp type from appearing more than once every 18,000 frames (~5 minutes). The UI layer binds chirp data as JSON for the frontend panel.

Out of scope: Life path event triggers that produce chirps, localization of chirp message text (see Localization).

How It Works

The Chirper system is a producer-consumer pipeline. Game systems (milestones, service issues, citizen events) enqueue ChirpCreationData requests into a shared NativeQueue. The CreateChirpSystem drains this queue each frame, resolves the chirp type, selects a sender, calculates engagement parameters, and creates the chirp entity. The ChirpLikeCountSystem then animates the like count over time, and the ChirperUISystem binds everything to the UI.

Three Sender Types

Every chirp has a sender, resolved in priority order:

  1. Service accounts -- City service chirps (water department, fire dept) use a dedicated ChirperAccountData entity as sender. The account has its own name, avatar, and links to an infoview.
  2. Brands -- Company/brand chirps use the company entity directly. The UI resolves the brand's logo as the avatar.
  3. Citizens -- Most chirps. The system tries to find a random living adult citizen from: the sender company's employees, the sender household's members, or (as fallback) any adult citizen in the city.

Deduplication (18,000-Frame Cooldown)

Before creating a chirp, the system checks if a chirp with the same prefab was created within the last 18,000 frames (~5 minutes at 60 FPS). If so, the request is silently dropped. This prevents repetitive "water shortage" or "crime wave" chirps from flooding the feed. The check is performed by CollectRecentChirpsJob, which builds a hashmap of (prefab Entity, chirp Entity) pairs from existing chirps.

Like Count Animation (Viral Curve)

Chirps don't instantly show their final like count. Instead, likes accumulate over time following a viral curve:

likes = targetLikes * (1 - (1 - progress)^viralFactor)

Where:

  • progress = fraction of time elapsed from creation to inactive frame
  • viralFactor = random integer 5-100 (higher = faster initial growth)
  • targetLikes = population * random factor (0.1%-3% of population)

With a high viral factor, likes spike quickly then plateau. With a low viral factor, growth is more linear. The m_ContinuousFactor (default 0.2) adds randomness by occasionally skipping update ticks.

Entity Name Caching

When a chirp references a citizen or building that later gets deleted (citizen dies, building demolished), the ChirpLinkSystem preserves the entity's name in a serializable cache. This ensures old chirps remain readable even after their referenced entities no longer exist.

  CreateChirpJob (Burst, IJob):
    Dequeue ChirpCreationData
    GetChirpPrefab: resolve trigger -> chirp prefab (random if multiple)
    Skip if recent chirp of same prefab exists (dedup)
    FindSender:
      Service -> ServiceChirpData.m_Account
      Brand -> sender entity directly
      Citizen -> random from company employees / household / population
    Calculate likes:
      targetLikes = population * random(0.001, 0.03)
      activeDuration = random(0.2, 1.0) * 262144 frames
      viralFactor = random(5, 100)
    Create entity with Chirp + PrefabRef + ChirpEntity buffer
          |
          v
LIKE ANIMATION (ChirpLikeCountSystem, every 64 frames)
  LikeCountUpdateJob (Burst, IJobChunk):
    progress = (currentFrame - creationFrame) / (inactiveFrame - creationFrame)
    likes = max(likes, targetLikes * (1 - (1-progress)^viralFactor))
    Random skip via continuousFactor for natural variation
          |
          v

Enqueue a Custom Chirp

Enqueue a chirp creation request from a mod system. CreateChirpSystem processes it on its next update.

public partial class CustomChirpTriggerSystem : GameSystemBase
{
    private CreateChirpSystem _createChirpSystem;

    protected override void OnCreate()
    {
        base.OnCreate();
        _createChirpSystem = World.GetOrCreateSystemManaged<CreateChirpSystem>();
    }

    protected override void OnUpdate()
    {
        if (!ShouldTriggerChirp()) return;

        JobHandle deps;
        NativeQueue<ChirpCreationData> queue = _createChirpSystem.GetQueue(out deps);
        deps.Complete();

        queue.Enqueue(new ChirpCreationData
        {
            m_TriggerPrefab = myTriggerPrefabEntity,  // Must have TriggerChirpData buffer
            m_Sender = senderHouseholdOrCompany,       // Household/company for citizen selection
            m_Target = Entity.Null                     // Optional linked entity
        });

        _createChirpSystem.AddQueueWriter(Dependency);
    }

    private bool ShouldTriggerChirp() { /* your logic */ return false; }
}

Read All Active Chirps

Query and log all current chirps in the game.

public void LogAllChirps(EntityManager em)
{
    EntityQuery chirpQuery = em.CreateEntityQuery(
        ComponentType.ReadOnly<Chirp>(),
        ComponentType.ReadOnly<PrefabRef>(),
        ComponentType.Exclude<Deleted>()
    );

    NativeArray<Entity> entities = chirpQuery.ToEntityArray(Allocator.Temp);
    NativeArray<Chirp> chirps = chirpQuery.ToComponentDataArray<Chirp>(Allocator.Temp);

    for (int i = 0; i < chirps.Length; i++)
    {
        Chirp chirp = chirps[i];
        Log.Info($"Chirp {entities[i]}: sender={chirp.m_Sender}, " +
                 $"likes={chirp.m_Likes}/{chirp.m_TargetLikes}, " +
                 $"viral={chirp.m_ViralFactor}, " +
                 $"liked={((chirp.m_Flags & ChirpFlags.Liked) != 0)}");
    }

    entities.Dispose();
    chirps.Dispose();
    chirpQuery.Dispose();
}

Modify Like Engagement via Harmony Postfix

Patch the like count system to apply a multiplier to all chirp engagement.

[HarmonyPatch(typeof(ChirpLikeCountSystem), "OnUpdate")]
public static class ChirpLikeMultiplierPatch
{
    private static float _likeMultiplier = 2.0f;

    public static void Postfix(ChirpLikeCountSystem __instance)
    {
        EntityManager em = __instance.EntityManager;
        EntityQuery query = em.CreateEntityQuery(
            ComponentType.ReadWrite<Chirp>(),
            ComponentType.Exclude<Deleted>()
        );

        NativeArray<Entity> entities = query.ToEntityArray(Allocator.Temp);
        for (int i = 0; i < entities.Length; i++)
        {
            Chirp chirp = em.GetComponentData<Chirp>(entities[i]);
            chirp.m_TargetLikes = (uint)(chirp.m_TargetLikes * _likeMultiplier);
            em.SetComponentData(entities[i], chirp);
        }

        entities.Dispose();
        query.Dispose();
    }
}

Filter Unwanted Chirps After Creation

Remove specific chirp types by deleting newly created chirp entities that match blocked prefabs.

public partial class ChirpCleanupSystem : GameSystemBase
{
    private EntityQuery _newChirpQuery;

    protected override void OnCreate()
    {
        base.OnCreate();
        _newChirpQuery = GetEntityQuery(
            ComponentType.ReadOnly<Chirp>(),
            ComponentType.ReadOnly<PrefabRef>(),
            ComponentType.ReadOnly<Created>(),
            ComponentType.Exclude<Deleted>()
        );
    }

    protected override void OnUpdate()
    {
        EntityManager em = EntityManager;
        NativeArray<Entity> entities = _newChirpQuery.ToEntityArray(Allocator.Temp);
        NativeArray<PrefabRef> prefabs =
            _newChirpQuery.ToComponentDataArray<PrefabRef>(Allocator.Temp);

        for (int i = 0; i < entities.Length; i++)
        {
            if (ShouldBlock(prefabs[i].m_Prefab))
            {
                em.AddComponent<Deleted>(entities[i]);
            }
        }

        entities.Dispose();
        prefabs.Dispose();
    }

    private bool ShouldBlock(Entity prefab) { /* your filter logic */ return false; }
}

Inspect the Chirp Queue API

Access the chirp creation queue to monitor or debug what systems are requesting chirps.

public void InspectChirpQueue(CreateChirpSystem createChirpSystem)
{
    // GetQueue returns the NativeQueue and a JobHandle for dependencies
    JobHandle deps;
    NativeQueue<ChirpCreationData> queue = createChirpSystem.GetQueue(out deps);
    deps.Complete();

    Log.Info($"Chirp queue count: {queue.Count}");

    // Important: after accessing the queue, register your dependency
    // so CreateChirpSystem waits for your access to complete
    createChirpSystem.AddQueueWriter(default(JobHandle));
}

Open Questions

  • Chirp localization: The RandomLocalizationIndex component selects a message variant via RandomLocalization.m_LocalizationID, but the full path from prefab to displayed text needs more investigation.
  • Trigger catalog: Many game systems enqueue ChirpCreationData (milestones, population thresholds, service issues), but the complete catalog of triggers is not documented.
  • Runtime chirp prefabs: Can modders register new chirp prefabs at runtime, or must they exist in the prefab system at load time? The TriggerChirpData buffer approach suggests load-time registration.
  • Burst job patching: The ChirpLikeCountSystem and CreateChirpSystem use Burst-compiled jobs. Harmony patches on Burst jobs require special handling (targeting the managed wrapper, not the Burst-compiled code).
